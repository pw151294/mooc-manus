## 任务总述
基于mooc-manus项目 `/api/agent/chat` 智能体对话核心主流程，落地**智能体工具调用死循环行为干预机制**，分两大核心产出物交付：
1. 结构化可落地的《功能开发任务契约》，拆解模块、数据结构、埋点逻辑、执行时序、代码接入点位；同时列出需我方确认的待澄清关键疑问项。
2. 标准化《功能全链路验证方案文档》，覆盖单元测试、集成测试、端到端自动化测试，端到端环节基于Playwright+MCP能力实现自动化用例编排与结果校验，定义干预机制生效与否的量化判定标准。

## 一、业务背景与问题定义（标准化书面描述）
### 现存缺陷
智能体执行工具调用时若单条工具调用持续执行失败，会反复重试发起**完全一致的工具调用请求**，形成工具调用死循环；
多轮失败日志持续追加至对话上下文尾部，会挤占上下文窗口权重，稀释大模型对对话上下文头部、中段任务目标、前置约束信息的注意力，进一步加剧无效重复调用，无法主动终止异常行为、上报故障或切换任务执行策略。

### 目标机制：工具调用行为干预熔断机制
在单条对话会话生命周期内，对重复失败的同源工具调用做次数计数，触达阈值后强制前置注入系统提示干预大模型推理，阻断无效循环调用，引导智能体上报异常、终止该工具调用分支或重新规划后续任务路径。

## 二、核心技术设计规范（精准约束实现逻辑）
### 1. 会话级内存计数器设计
1. 生命周期：绑定单次Agent Chat对话会话，会话销毁则计数器数据自动回收，不做持久化存储。
2. 键（Key）唯一标识规则：`工具名称 + 调用入参指纹哈希值`，确保仅**完全相同工具+完全相同参数**的调用被归为同一条失败项计数；参数微小变更则视为新调用条目，独立统计。
3. 值（Value）：该Key对应调用记录的连续失败累计次数。

### 2. 计数更新时机
每一轮大模型推理输出、智能体完成一轮工具调用执行并判定调用结果为失败后，立即检索并更新该条目标Key对应的失败计数器。

### 3. 阈值与干预触发时机
1. 预设熔断阈值：单条目累计失败次数 ≥ 3 即触发行为干预；
2. 触发节点：**下一轮向大模型发起Prompt请求之前**执行前置干预，不可在LLM请求途中或返回后后置处理。

### 4. 干预注入方式与Prompt设计要求
以**用户侧系统提示词**追加注入上下文头部，向大模型明确传递三条约束：
1. 告知当前某一工具+对应参数组合已连续失败达阈值，禁止本轮及后续轮次再次发起完全一致的工具调用；
2. 要求智能体二选一执行后续逻辑：①向上反馈该工具调用失败原因与故障信息，结束该子任务；②基于已有上下文重新拆解任务，更换工具、修改入参后再发起新调用；
3. 禁止模型忽略该干预提示继续循环重试同源调用。

### 交付项1：《行为干预熔断机制·开发任务契约》
结构必须包含以下章节：
1. 需求边界与不做事项（明确机制生效范围、排除场景）
2. 核心数据结构定义（内存计数器结构体/Map结构、哈希生成逻辑）
3. 主流程接入点位：定位`/api/agent/chat`链路中：会话初始化、工具调用结果回执、计数更新、阈值判断、Prompt注入、会话结束销毁计数器 共5个关键代码埋点位置
4. 分层开发子任务拆解（可按模块拆分：会话上下文模块、工具调用结果拦截器、计数管理模块、干预Prompt拼接模块、会话生命周期钩子）
5. 异常分支处理（计数器溢出、哈希计算异常、并发会话隔离、多工具批量失败场景）
6. 需人工确认澄清的关键问题清单（逐条列出，每条标注影响开发方案的原因）

### 交付项2：《全维度测试验证规范文档》
整体分为三层测试体系，同时定义**干预有效性强判定标准**（可量化，避免主观校验）：
#### （1）单元测试
- 测试对象：哈希指纹生成函数、计数器增删查改、阈值判定逻辑、干预Prompt模板渲染
- 必覆盖用例：正常计数累加、阈值临界值、参数变更新建计数条目、会话销毁清空数据、重复Key重复计数

#### （2）集成测试
- 测试链路：模拟单轮对话→工具调用失败→计数+1；循环至第3次失败；校验前置Prompt是否成功注入LLM入参上下文；
- 校验点：上下文是否携带干预提示、模型输出是否不再生成同源工具调用指令

#### （3）端到端自动化E2E测试
1. 基于Playwright搭建自动化测试脚本，依托MCP工具能力完成接口调用、会话创建、消息发送、响应监听；
2. 构造标准化死循环复现场景：固定入参工具强制返回失败，观测智能体前3轮重复调用，第4轮被机制拦截；
3. 自动化断言规则：
    - 断言计数器数值准确；
    - 断言第4轮LLM入参包含干预提示文本；
    - 断言后续返回结果无同源工具调用Action；
4. 额外补充反向用例：参数修改后允许新建计数，不会被旧条目熔断拦截。

#### （4）干预机制有效性验收基线
明确可落地的Pass/Fail判定条件，用于上线准入校验。

## 四、额外约束
所有方案必须贴合mooc-manus现有项目架构与`/api/agent/chat`原有执行链路，尽量最小侵入改造，优先采用钩子/中间件/结果拦截器方式接入，禁止大规模重构原有主逻辑；代码方案需兼容现有SSE流式输出、对话上下文管理、多智能体协同架构。

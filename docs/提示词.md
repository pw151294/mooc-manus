请你为 mooc-manus 编程智能体完整设计并实现一套自动化评测机制，包含完整方案设计与代码落地，需求规范如下：

## 1. 开发背景
当前 mooc-manus 智能体已具备完整编程智能体能力与全链路追踪能力，可记录智能体每一步执行过程。为量化智能体任务完成准确率、运行稳定性、资源消耗与执行效率，指导智能体迭代优化，需要搭建一套标准化、可持久化、可批量执行、可观测的自动化评测体系。本次评测范围聚焦智能体核心工具能力：文件读写、文件编辑、bash 脚本执行。

## 2. 数据层设计（Repo 持久化模型）
整体分为四张核心数据表：评测用例、评测任务、评测运行实例、评测结果，完整覆盖评测全生命周期数据。

### 2.1 评测用例
用于定义标准化评测场景，字段包含：用例名称、可选初始化脚本、任务指令、验证脚本。
- 初始化脚本：用于沙盒环境前置环境初始化，保证评测环境隔离一致
- 任务指令：给到智能体的评测 prompt
- 验证脚本：执行后通过 exit_code=0 判定评测通过，非 0 判定失败

### 2.2 评测任务
由用户前端创建，为批量评测父级任务，包含任务名称、任务整体状态机，需要你设计完整状态枚举与合法状态流转，用于展示整体评测进度。

### 2.3 评测运行实例
评测最小执行单元，支持 M 个评测用例 * N 个智能体配置批量评测，创建评测任务时自动批量生成 M*N 条运行实例。
字段包含：关联评测任务唯一ID、关联评测用例唯一ID、关联智能体应用配置唯一ID、独立实例状态机。
需要单独设计实例级状态机，细粒度跟踪每条评测的执行状态。

### 2.4 评测结果
单条运行实例执行结束后自动生成并持久化，字段包含：关联运行实例ID、评测是否通过（布尔）、错误日志、token 消耗总量、任务执行耗时。
- token 消耗、执行耗时从现有链路追踪 span 扩展标签中解析、汇总、统计

## 3. API 接口层能力
需要向前端管理平台暴露完整评测管理接口：

### 3.1 评测用例上传创建
支持用户创建评测用例，每一项内容（初始化脚本/任务指令/验证脚本）支持「手动输入」或「文件上传解析」两种方式，单项二选一互斥，不可同时填充。

### 3.2 评测任务创建
用户选择批量评测用例、批量智能体模型配置，后台自动完成 M*N 批量实例生成，创建一条父任务 + M*N 条子运行实例。

### 3.3 状态查询
支持分页、多条件筛选：查询评测任务列表、查询任务下所有运行实例列表，支持状态筛选、进度查看。

### 3.4 重试与删除
支持整任务重试：仅重试任务下失败/异常实例；
支持单运行实例独立重试；
支持删除整条评测任务、删除单个运行实例。

## 4. 领域层核心执行流程
1. 根据评测运行实例拉取对应评测用例配置
2. 执行前置初始化脚本，初始化沙盒环境
3. 根据智能体配置ID初始化对应智能体
4. 组装评测对话请求，参考项目内部 StreamingInvoke 异步启动智能体任务
5. 阻塞等待智能体执行完毕
6. 执行验证脚本，根据 exit_code 判断结果
7. 解析链路追踪 span，统计总耗时、总 token 消耗
8. 生成评测结果并入库持久化

同时实现全局定时巡检任务，定时刷新评测任务、评测运行实例状态，保证前端状态与实际执行状态一致。

## 5. 需要你重点攻克的技术难点并输出完整方案
1. **双状态机设计**：分别设计评测任务、评测运行实例两套状态枚举与完整流转规则，包含待执行、执行中、成功、失败、重试中、已终止等状态，定义全部合法流转关系。
2. **评测指标采集逻辑**：实现安全的验证脚本执行与结果判分、基于链路追踪 span 的 token 统计、耗时统计算法。
3. **高并发批量评测架构**：禁止直接批量开 goroutine，需要引入消息队列进行任务解耦。请完整设计：消息队列选型、topic 设计、消息体结构、消费策略、分区策略、限流削峰、消息持久化、宕机恢复、防重复消费、失败重试机制，解决批量评测 goroutine 泄露与并发爆炸问题。
